Les views — la logique métier

JWT — comment fonctionne l'authentification

JWT (JSON Web Token) est un système de badge numérique. Quand tu te connectes, le serveur te donne un token que tu présentes à chaque requête.

1

Login

Tu envoies email + mot de passe à /api/auth/login/. Le serveur vérifie et te renvoie deux tokens.

POST /api/auth/login/ {"username": "marietou", "password": "..."}
2

Réception des tokens

Le serveur renvoie un access_token (valide 30 min) et un refresh_token (valide 7 jours).

{"access": "eyJhbG...", "refresh": "eyJhbG..."}
3

Utilisation de l'API

À chaque requête, tu joins l'access_token dans le header. Sans ça → erreur 401.

GET /api/patients/ Authorization: Bearer eyJhbG...
4

Token expiré → refresh

Après 30 min, l'access_token expire. Tu envoies le refresh_token pour en obtenir un nouveau sans te reconnecter.

POST /api/auth/refresh/ {"refresh": "eyJhbG..."}

Structure d'un token JWT

eyJhbGciOiJIUzI1NiJ9 . eyJ1c2VySWQiOjEsInJvbGUiOiJtZWRlY2luIn0 . SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c

Les tests — pourquoi et comment

Un test vérifie qu'une partie de ton code fait exactement ce qu'elle doit faire. En entreprise, le code sans tests n'est pas accepté.

Django (Python)

class PatientTest(TestCase): def setUp(self): self.patient = Patient.objects.create( nom="Diop", prenom="Fatou", ... ) def test_creation(self): self.assertEqual(self.patient.nom, "Diop")

Spring Boot (Java)

@SpringBootTest class PatientTest { @BeforeEach void setUp() { patient = new Patient("Diop", ...); } @Test void testCreation() { assertEquals("Diop", patient.getNom()); } }

Équivalences

@BeforeEach → setUp(self)
@Test → def test_xxx(self)
assertEquals → self.assertEqual
assertTrue → self.assertTrue
MockMvc → APIClient()
L'erreur que tu as eu : self.Patient (P majuscule) ≠ self.patient (p minuscule). Python est sensible à la casse — ce sont deux variables différentes. setUp crée self.patient mais les tests cherchaient self.Patient.
1

Test de modèle

Vérifie que l'objet est créé correctement, que les valeurs par défaut sont bonnes.

2

Test d'API

Vérifie que les endpoints renvoient les bons codes HTTP (200, 201, 401, 404).

3

Test d'authentification

Vérifie qu'un utilisateur sans token reçoit bien une erreur 401.
